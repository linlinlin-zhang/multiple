# Phase 26 Plan 01 — Responses API Migration

## Goal

Migrate the canvas chat system from DashScope Chat Completions to DashScope Responses API while preserving the existing canvas tool-calling UX and preparing the UI for citations and artifacts.

## Requirements Covered

- RESP-01: `/api/chat` uses DashScope Responses API endpoint.
- RESP-02: builtin `web_search` coexists with custom `canvas_action` function calls.
- RESP-03: `code_interpreter` is included when code/data-analysis intent is detected.
- RESP-04: `web_extractor` is included when URL/web-page extraction intent is detected.
- RESP-05: `[ref_<n>]` citations are returned as structured references for clickable rendering.
- RESP-06: code/interpreter outputs are normalized as chat artifacts for preview/download.
- RESP-07: Responses `previous_response_id` is returned to the client and sent on follow-up turns.

## Current Architecture

- `server.js` owns `/api/chat`, model configuration, tool schemas, Chat Completions calls, SSE streaming, fallback action inference, reference extraction, and RAG injection.
- `public/app.js` owns the chat sidebar, `postStreamingChat()`, markdown rendering, action execution, artifacts, chat threads, and session persistence.
- `qwenResponsesRequest()` and `qwenResponsesBaseUrl()` already exist in `server.js`, but are only used by image search.
- `CANVAS_ACTION_TOOL_SCHEMA` is currently OpenAI Chat Completions shaped and must be adapted for Responses API without breaking existing extraction helpers.

## Implementation Tasks

1. Add Responses API request builders and response normalizers in `server.js`.
2. Change `/api/chat` to build Responses input instead of `messages[]` for chat turns.
3. Include builtin tools conditionally: `web_search`, `web_extractor`, `code_interpreter`, plus custom `canvas_action`.
4. Normalize Responses function calls into the same action format used by `applyVoiceActions()`.
5. Normalize citations and artifacts into frontend-friendly arrays.
6. Stream Responses text deltas through the existing `reply` SSE event and preserve final payload shape.
7. Persist and reuse `previous_response_id` in frontend chat thread state.
8. Add minimal citation/artifact rendering polish in `public/app.js`.

## Verification

- Server syntax check passes.
- `/api/chat` demo mode still works without API keys.
- Streaming final payload includes `responseId` when the Responses API returns an id.
- A web-search question can produce references/artifacts without breaking canvas actions.
- A canvas command such as “做个日本 7 天计划” still returns executable `canvas_action` actions.

## Risks

- DashScope Responses event names may vary; stream parser must accept multiple text/function-call shapes.
- Builtin tool output schemas may be nested or stringified; extraction should recursively scan JSON.
- Previous response IDs should be per chat thread, not global across the whole session.
