# Phase 26 Plan 04 — Universal AI App Answer Quality Upgrade

## Goal

Upgrade ORYZAE from a tool-first canvas assistant into a more general-purpose AI application with stronger breadth, depth, and answer quality while keeping canvas actions, Responses API built-in tools, citations, artifacts, and multi-turn continuity.

## Research Notes

This plan is based on observable public product patterns and public documentation. It does not attempt to copy hidden system prompts from ChatGPT, Qwen, Claude, or other official apps.

### Patterns from advanced official AI apps

1. Chat remains the primary answer surface. Tools and artifacts augment the answer; they do not replace the answer with a short confirmation.
2. Complex deliverables are self-contained and reusable. Claude artifacts are created for significant standalone content that users may edit, iterate on, export, or revisit.
3. Server-side tools are visible in the response object, but the final assistant message should synthesize tool results into a useful answer.
4. Web search should produce source-grounded synthesis with citations, not only URL cards.
5. Code/data tools should return conclusions, caveats, and outputs, not only execution traces.
6. Long or complex tasks need larger output budgets and graceful handling of truncation.
7. Context management matters: preserve multi-turn continuity, but compact noisy UI/canvas context so the model can focus on the user request.
8. Tool schemas should be rich enough to hold useful structured output, but prompts must still require a readable chat summary.

### Current ORYZAE bottlenecks

1. The chat system prompt says the rich node card is the deliverable, which encourages short chat confirmations.
2. `synthesizeReplyFromActions()` returns short operation acknowledgements when tool calls dominate.
3. `create_plan` schema only documents minimal `steps.title/description`, so travel/planning cards tend to be shallow.
4. The prompt packs large raw canvas/state JSON and lacks an explicit answer-quality contract.
5. Request options only pass configured `max_tokens`; task-specific output budgets are not applied.
6. Web/tool use is wired, but final synthesis quality is not explicitly enforced.

## Design Contract

### Answer quality profile

For normal non-trivial requests, ORYZAE should:

- Answer directly first.
- Expand with relevant structure, examples, tradeoffs, caveats, and next steps.
- Match the user's language.
- Use markdown headings/lists for readability.
- Ask clarifying questions only when necessary; otherwise make reasonable assumptions and state them.
- Avoid artificially short replies unless the user explicitly asks for brevity.

### Tool and canvas contract

- Use canvas actions only when they materially improve the result or the user asks for canvas output.
- When using canvas actions, still write a substantive chat answer that summarizes the key content and how to use the created card.
- Do not let action templates replace a substantive answer unless the action is a simple UI command.
- Rich nodes should be self-contained enough to reuse outside the chat.

### Context contract

- Keep previous_response_id for multi-turn continuity.
- Include recent messages when no previous_response_id exists.
- Summarize or cap canvas state instead of overloading the model with irrelevant raw JSON.

### Request budget contract

- Apply larger `max_tokens` to planning, research, analysis, comparison, tutorial, writing, and code/data tasks.
- Keep short prompts inexpensive for greetings and simple commands.
- Respect user-configured settings when they are higher than the task-specific defaults.

## Implementation Plan

1. Add an explicit universal answer-quality section to `buildChatSystemContext()`.
2. Add task-specific response guidance to `buildChatUserPrompt()`.
3. Make canvas-tool instructions say "chat answer plus canvas artifact" instead of "card is the deliverable".
4. Expand `canvas_action.content` documentation for richer `create_plan`, `create_note`, `create_web_card`, and `create_code` payloads while preserving existing renderer compatibility.
5. Add server-side task complexity detection and task-specific `max_tokens` defaults.
6. Replace terse synthesized action replies with richer summaries derived from action content when the model does not provide text.
7. Compact canvas context before sending it to the model.
8. Verify through API smoke tests and UI checks.

Streaming abort and table overflow fix:

- Observed issue: long streamed report initially rendered normally, then the request aborted and the frontend replaced the whole assistant message with "This operation was aborted"; no canvas cards appeared because no `final` SSE event arrived.
- Root cause: frontend catch logic overwrote the pending assistant message with the error text, discarding already-streamed content.
- Root cause: long streaming calls used a fixed timeout instead of an idle timeout refreshed by stream activity, so lengthy but active responses could be aborted.
- Root cause: pending chat text used `inline-flex`, and markdown tables were allowed to influence message width during streaming.
- Added `CHAT_STREAM_IDLE_TIMEOUT_MS` with a larger default and refreshed the timeout when Responses/Chat streaming chunks arrive.
- Changed frontend streaming error handling to preserve partial reply content and append an interruption note instead of replacing the whole message.
- Marked partial errors as handled to avoid double-appending interruption messages.
- Tightened chat markdown layout: assistant messages now have `min-width: 0`, pending text no longer uses flex layout, chat text is bounded to `max-width: 100%`, and markdown table cells wrap with a per-cell max width while the table scrolls horizontally inside the message.
