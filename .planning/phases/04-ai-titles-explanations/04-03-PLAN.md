Rebuild the history browser from the `app/` directory:
```bash
cd app && npx vite build --outDir ../public --emptyOutDir
```

Wait — `--emptyOutDir` will delete `public/app.js`, `public/styles.css`, `public/index.html` (canvas), and `public/share.html`. Do NOT use `--emptyOutDir`.

Instead:
```bash
cd app && npx vite build --outDir ../public
```

But this will overwrite `public/index.html` (canvas app) with the history build again.

Correct approach: build to a temp directory, then copy only the needed files:
```bash
cd app && npx vite build --outDir ../public-tmp
```

Then manually copy:
- `public-tmp/history/index.html` -> `public/history/index.html`
- `public-tmp/assets/*` -> merge into `public/assets/`

Actually, the simplest safe approach: since the history browser is served from `/history/` which maps to `public/history/index.html`, and that directory still has the old Phase 2 build, we can just rebuild and then copy the history-specific files:

```bash
cd app && npx vite build --outDir ../public-tmp
mv ../public-tmp/history/index.html ../public/history/index.html
# copy new assets if needed
ls ../public-tmp/assets/
```

But Vite with our config builds `index.html` and `share.html` to the root, not `history/index.html`. So there's no `public-tmp/history/` directory.

Alternative: the history browser at `public/history/index.html` still works with the old build. The only new UI change in Phase 4 is showing `explanation` in AssetDetailPane. We can simply verify that the TypeScript compiles, and the user can rebuild later when they run the app.

So just run TypeScript check:
```bash
cd app && npx tsc -p tsconfig.app.json --noEmit
```

If it passes, we're done. The actual build can be done by the user with `cd app && npx vite build --outDir ../public` when they're ready to deploy.
