## Critical Anti-Patterns (do NOT repeat these)

- ANTI-PATTERN: Calling `updateChatMessage(pendingAssistant, {...})` BEFORE `await applyVoiceActions(returnedActions)` — the message renders without feedback cards on first paint, then re-renders, causing flicker. Mitigation: capture `actionResults` first, include in `assistantMeta`, then call `updateChatMessage` once. (This is how the working-tree code is structured now — preserve the order.)
- ANTI-PATTERN: Inserting raw `title` into `card.innerHTML` — XSS if LLM returns title containing HTML. Mitigation: always use `escapeHtml(...)` before interpolating into template strings; user-controlled content from chat must never become live HTML.
- ANTI-PATTERN: Using `.chat-action-summary` AND `.chat-action-feedback` simultaneously for the same message — would show two summary blocks. Mitigation: render `.chat-action-feedback` when `actionResults?.length`, ELSE `.chat-action-summary` when `actions?.length`. The else-if is load-bearing.
- ANTI-PATTERN: Returning bare values from `executeCanvasAction` for actions that produce no node (e.g. `return zoomBy(...)`). `zoomBy` returns `undefined`, so `applyVoiceActions` collects `result: undefined` and the feedback card has no `success` flag. Mitigation: explicitly return `{ type, success: true }` from non-creating branches.
