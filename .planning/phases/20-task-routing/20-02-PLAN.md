# Plan 02: Frontend Task-Type Badges & Template Routing

## Goal
Display task-type badges on direction cards and route each task type to its specialized prompt template for qualitatively different outputs.

## must_haves
- truths:
  - Badges must not break existing card layout or overlap with other UI elements
  - Template routing must be backward-compatible: unknown task types fall back to default analysis
  - New prompt templates must reuse existing `src/prompts/` builders, not duplicate logic

## Tasks

### Task 1: Extend prompt builders with task-type bias

<read_first>
- src/prompts/analysis.js (buildAnalysisPrompt, buildExplorePrompt)
- src/prompts/index.js (exports)
</read_first>

<action>
Extend `buildAnalysisPrompt(lang, taskType = 'general')` and `buildExplorePrompt(lang, taskType = 'general')` in `src/prompts/analysis.js`:

1. When `taskType === 'research'`: Add an extra instruction in the prompt asking the LLM to prioritize `research` and `content` purpose options over `visual`
2. When `taskType === 'planning'`: Add an extra instruction asking the LLM to prioritize `plan` and `tool` purpose options
3. When `taskType === 'creative'`: Add an extra instruction asking the LLM to prioritize `visual` and `exploration` purpose options
4. When `taskType === 'image_generation'`: Same as existing behavior (default visual bias)
5. When `taskType === 'general'` or unknown: No change to existing behavior

<acceptance_criteria>
- buildAnalysisPrompt accepts an optional second parameter `taskType` with default 'general'
- buildExplorePrompt accepts an optional second parameter `taskType` with default 'general'
- For taskType 'research', the prompt contains text asking to prioritize research directions
- Existing calls without the second parameter behave identically to before
</acceptance_criteria>

### Task 3: Wire task-type into analysis flow

<read_first>
- public/app.js (where analysis results are received and cards are created)
- server.js (handleAnalyze response format)
</read_first>

<action>
1. In `server.js`, ensure `handleAnalyze` and `handleAnalyzeExplore` include `taskType` in the JSON response alongside `title`, `summary`, `options`
2. In `public/app.js`, when processing analysis response, pass `response.taskType || 'general'` to card creation
3. When the user clicks a direction card, the `taskType` is stored in the card's data attributes for future reference

<acceptance_criteria>
- Server analysis responses include a `taskType` field
- Frontend passes taskType from response into card creation
- Card data attributes include `data-task-type`
</acceptance_criteria>
