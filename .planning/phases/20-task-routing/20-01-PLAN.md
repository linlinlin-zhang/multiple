# Plan 01: Task Classification API & Routing Engine

## Goal
Build a `POST /api/route-task` endpoint that uses LLM to classify uploaded content into task types and routes to the appropriate prompt template.

## must_haves
- truths:
  - Classification must use the existing LLM chat configuration (runtimeConfigs.chat)
  - Low-confidence results fall back to file-format-based routing (image → visual, pdf/doc → research, etc.)
  - All new code must be in ES modules, zero new runtime dependencies
  - The router must be reusable by both analyze and explore handlers

## Tasks

### Task 1: Create task router module

<read_first>
- src/prompts/index.js (existing prompt exports)
- src/prompts/analysis.js (buildAnalysisPrompt structure)
- server.js (handleAnalyze, handleExplore handlers for context)
</read_first>

<action>
Create `src/lib/taskRouter.js` with:

1. `classifyContent({ content, contentType, fileName, lang })` — async function that:
   - Builds a classification prompt asking the LLM to choose from: `image_generation`, `research`, `planning`, `creative`, `content`, `tool`, `general`
   - Includes the first 2000 chars of content (or image metadata) in the prompt
   - Calls the existing chat completions API via `chatCompletions(config, payload)`
   - Returns `{ taskType, confidence: 0-1, rationale }`

2. `getFallbackTaskType({ contentType, fileName })` — sync function:
   - Maps MIME types and extensions to default task types
   - `image/*` → `image_generation`
   - `application/pdf`, `text/*`, `application/vnd.*` → `research`
   - `audio/*`, `video/*` → `content`
   - Unknown → `general`

3. `resolveTaskType(classificationResult, fallbackResult, threshold = 0.6)` — sync function:
   - If classification.confidence >= threshold → return classification.taskType
   - Else → return fallback.taskType with `wasFallback: true`

4. `getPromptForTaskType(taskType, lang)` — sync function:
   - Maps task types to prompt builder functions imported from `src/prompts/index.js`
   - `image_generation` → `buildAnalysisPrompt(lang)`
   - `research` → `buildExplorePrompt(lang)` or a new research-focused prompt
   - `planning` → `buildAnalysisPrompt(lang)` with planning bias
   - `creative` → `buildAnalysisPrompt(lang)` with creative bias
   - `general` → `buildAnalysisPrompt(lang)`

<acceptance_criteria>
- `src/lib/taskRouter.js` exists and exports `classifyContent`, `getFallbackTaskType`, `resolveTaskType`, `getPromptForTaskType`
- `node -c src/lib/taskRouter.js` passes syntax check
- The module imports at least 3 prompt builders from `../prompts/index.js`
</acceptance_criteria>

### Task 2: Add POST /api/route-task endpoint

<read_first>
- server.js (existing route handlers, esp. handleAnalyze and handleExplore)
- src/lib/taskRouter.js (from Task 1)
</read_first>

<action>
In `server.js`, add `handleRouteTask(req, res)` async function:

1. Parse body: `{ content, contentType, fileName, language, useLLM = true }`
2. If `useLLM` is true:
   - Call `classifyContent({ content, contentType, fileName, lang })`
   - Call `getFallbackTaskType({ contentType, fileName })`
   - Call `resolveTaskType(classification, fallback)`
3. If `useLLM` is false:
   - Skip LLM call, use fallback directly
4. Return JSON:
   ```json
   {
     "taskType": "research",
     "confidence": 0.87,
     "wasFallback": false,
     "rationale": "The document contains structured questions and reference requests..."
   }
   ```
5. Wire route: `POST /api/route-task` → `handleRouteTask`

<acceptance_criteria>
- `POST /api/route-task` returns 200 with JSON containing `taskType`, `confidence`, `wasFallback`, `rationale`
- When `useLLM=false`, confidence is 0 and wasFallback is true
- `node -c server.js` passes syntax check
</acceptance_criteria>

### Task 3: Wire task routing into analyze handlers

<read_first>
- server.js (handleAnalyze, handleAnalyzeExplore, handleAnalyzeUrl, handleAnalyzeText)
- src/lib/taskRouter.js
</read_first>

<action>
Modify `handleAnalyze` and `handleAnalyzeExplore` in `server.js`:

1. After extracting `lang` and before building the analysis payload, call the task router:
   ```javascript
   const { taskType, confidence, wasFallback } = await routeContent({
     content: extractedText || imageDataUrl || url,
     contentType: body.contentType || (imageDataUrl ? "image" : "text"),
     fileName: body.fileName,
     lang
   });
   ```
2. Pass `taskType` into the prompt builder (extend builders to accept optional `taskType` bias)
3. Log routing decision for debugging: `[route] ${taskType} (confidence: ${confidence}, fallback: ${wasFallback})`

<acceptance_criteria>
- server.js analysis handlers call routeContent before building prompts
- Console logs include `[route]` prefix with task type and confidence
- Existing analyze/explore behavior is preserved when taskType is "general"
</acceptance_criteria>
