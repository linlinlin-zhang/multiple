  <action>
Create `src/lib/textExtract.js` that exports `extractTextFromBuffer(buffer, ext)`.

Supported formats and extraction strategy:
1. **TXT / MD / JSON / plain text** (`ext` in `txt,md,json`):
   - Return `buffer.toString("utf8")`.
   - If decoding fails (non-UTF8), try `buffer.toString("latin1")`.
   - Strip BOM if present.

2. **DOCX** (`ext === "docx"`):
   - DOCX is a ZIP archive containing `word/document.xml`.
   - Use Node.js built-in `node:zlib` to inflate the XML (or treat as ZIP).
   - Actually, Node 18 has no built-in ZIP extraction. Use a minimal approach:
     - Read the buffer, search for text runs between `<w:t>` and `</w:t>` tags.
     - Concatenate matches with spaces.
   - This is a heuristic but works for most DOCX files without adding dependencies.
   - Cap extracted text at 8000 characters to avoid huge prompts.

3. **PDF** (`ext === "pdf"`):
   - PDF text extraction without a library is limited. Use a regex heuristic:
     - Search for text between `( ... )` sequences that are not image data.
     - Filter out sequences that are mostly binary / hex.
     - Alternatively, look for `BT ... ET` blocks and extract text after `Tj` or `TJ` operators.
   - Cap at 8000 characters.
   - Document the limitation: "PDF extraction is heuristic; complex layouts may yield partial text."

4. **PPT / PPTX** (`ext` in `ppt,pptx`):
   - For PPTX (ZIP), similar to DOCX: search `ppt/slides/*.xml` for `<a:t>` text elements.
   - For old PPT (binary), skip with a clear error: "Legacy .ppt format not supported; please convert to .pptx."
   - Cap at 8000 characters.

All extractors:
- Trim whitespace.
- Collapse multiple whitespace to single spaces.
- Return `{ text: string, truncated: boolean }`.
- If format is unsupported, throw `Error("Unsupported file format: .${ext}")`.

No external npm dependencies — use only Node.js built-ins (Buffer, zlib for gzip/deflate if needed).
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/analyze-text | Untrusted file upload crosses here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | /api/analyze-text request body | mitigate | Validate body size (reuse MAX_BODY_BYTES), reject non-string text, sanitize fileName with `slug()` |
| T-06-02 | Information Disclosure | Extracted text sent to AI API | accept | Text is user-owned content sent for analysis; no PII expected in creative workflow. Documented limitation. |
| T-06-03 | Denial of Service | Large file upload | mitigate | MAX_BODY_BYTES (22MB) already enforced by readJson; extraction caps text at 8000 chars before prompt |
| T-06-04 | Elevation of Privilege | File path traversal via fileName | mitigate | `slug()` sanitizes fileName; storage uses SHA-256 hash paths, never user-provided names |
| T-06-05 | Spoofing | Fake dataUrl mime type | mitigate | Derive extension from fileName (trusted client-side File.name) rather than dataUrl mime type for extraction routing |
</threat_model>
